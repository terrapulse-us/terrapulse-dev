1. MapLibre Tile (MLT) via OPFS
MVT is being actively replaced by the MapLibre Tile (MLT) format, which was designed from the ground up for modern hardware APIs.

Columnar Storage: MLT uses a Decomposition Storage Model (column-oriented). This achieves up to a 6x compression ratio improvement over MVT on complex, dense vector layers (like topographic contours).

Direct-to-GPU: Because the data is columnar, it can be decoded using SIMD instructions directly into GPU buffers. You bypass the CPU parsing bottleneck completely.

OPFS Persistence: Instead of running a heavy local SQLite database, modern progressive and hybrid apps store these single-file archives directly in the device's Origin Private File System (OPFS), allowing instantaneous byte-range reads with zero database overhead.

The Result: This completely eliminates the concept of "zoom levels." The map remains perfectly sharp from a global view down to millimeter precision.

Inside the WebView

import * as duckdb from '@duckdb/duckdb-wasm';

async function initDatabase() {
  // 1. Point DuckDB to the local files bundled in your app
  const bundles = {
    mvp: {
      mainModule: 'local_assets/duckdb.wasm',
      mainWorker: 'local_assets/duckdb-worker.js',
    }
  };
  
  const logger = new duckdb.ConsoleLogger();
  const worker = new Worker(bundles.mvp.mainWorker);
  const db = new duckdb.AsyncDuckDB(logger, worker);
  
  await db.instantiate(bundles.mvp.mainModule);
  
  // 2. Load the local spatial extension
  const conn = await db.connect();
  await conn.query(`INSTALL 'local_assets/spatial.duckdb_extension'; LOAD spatial;`);

  // 3. Register the GeoParquet file stored in OPFS so DuckDB can read it
  const opfsRoot = await navigator.storage.getDirectory();
  const fileHandle = await opfsRoot.getFileHandle('desert_trails.parquet');
  await db.registerFileHandle('desert_trails.parquet', fileHandle, duckdb.DuckDBDataProtocol.BROWSER_FILEREADER, true);
  
  return conn;
}
4
Wire MapLibre's camera to spatial queries
The bounding box loop
You don't want to render the whole desert at once. You hook into MapLibre's moveend event to get the current screen coordinates and pass them to DuckDB.

JavaScript
map.on('moveend', async () => {
  // 1. Get the current map boundaries
  const bounds = map.getBounds();
  const minX = bounds.getWest();
  const minY = bounds.getSouth();
  const maxX = bounds.getEast();
  const maxY = bounds.getNorth();

  // 2. Query DuckDB for features strictly inside this box
  const query = `
    SELECT ST_AsGeoJSON(geom) AS geometry, trail_name, difficulty
    FROM read_parquet('desert_trails.parquet')
    -- Spatial index check for instant lookup
    WHERE ST_Intersects(
      geom, 
      ST_MakeEnvelope(${minX}, ${minY}, ${maxX}, ${maxY})
    )
  `;

  // 3. Execute the query
  const arrowResult = await conn.query(query);
  
  // 4. Convert Arrow tabular data back into standard GeoJSON for MapLibre
  const features = arrowResult.toArray().map((row) => ({
    type: 'Feature',
    geometry: JSON.parse(row.geometry),
    properties: { name: row.trail_name, difficulty: row.difficulty }
  }));

  const geoJsonFeatureCollection = { type: 'FeatureCollection', features };
  
  // 5. Stream to the map (See next step)
  updateMapData(geoJsonFeatureCollection);
});

To achieve lightning-fast reads on a mobile device, you have to optimize for how DuckDB-WASM interacts with the Origin Private File System (OPFS).

When the user pans the map, DuckDB doesn't load the whole file. It reads the metadata header, finds the "Row Groups" that overlap the current screen coordinates, and extracts only those specific byte ranges. If your data is scattered randomly throughout the file, DuckDB has to download almost everything, destroying performance.

To prevent this, you must perfectly cluster your data on disk. Here is the ultimate DuckDB desktop optimization pipeline to prepare your files for mobile.

1
Load the spatial extension
DuckDB CLI
Before running any geographic commands, you must load the spatial extension. If you forget this, DuckDB will write the geometry as useless binary blobs instead of valid GeoParquet.

SQL
INSTALL spatial; 
LOAD spatial;
2
Execute the optimized COPY command
The master query
You can process and optimize your raw data (like a GeoJSON or Shapefile) into a mobile-ready GeoParquet file in a single, powerful SQL query:

SQL
COPY (
  SELECT *
  FROM st_read('desert_trails.geojson')
  ORDER BY ST_Hilbert(
    geom, 
    -- Dynamically calculate the bounding box of your specific dataset
    (SELECT st_extent(ST_Extent_Agg(COLUMNS(geom)))::BOX_2D FROM st_read('desert_trails.geojson'))
  )
) TO 'optimized_trails.parquet' (
  FORMAT PARQUET,
  COMPRESSION 'zstd',
  ROW_GROUP_SIZE 100000
);

Why this specific configuration works
There are three critical optimization levers in that query that make it perfect for mobile OPFS:

ST_Hilbert with Bounds: This maps 2D geometries to a 1D index, physically storing trails that are next to each other in the real world next to each other on the SSD. A crucial mistake developers make is omitting the bounding box argument. Passing the dynamic bounding box ensures the Hilbert curve wraps tightly around your specific off-roading region, rather than stretching across the entire planet.

COMPRESSION 'zstd': DuckDB writes Parquet files using Snappy compression by default. By explicitly switching to zstd, you will shrink your final payload by an additional 20% to 30%, which saves space on the user's phone with zero penalty to read speed.

ROW_GROUP_SIZE 100000: Parquet files are split into chunks called Row Groups. When the user pans the map, DuckDB looks at the bounding box for each row group and skips the ones outside the screen. If your row groups are too large, DuckDB reads too much useless data. If they are too small, the metadata overhead bloats the file. The established sweet spot for spatial queries is 50,000 to 100,000 rows per group.

(Note: If you prefer command-line tools over raw SQL, there is an open-source Python tool called geoparquet-io (gpio) that automatically applies these exact DuckDB optimizations by default.)

1
Partition by Geohash Grid
DuckDB CLI
This query takes a massive master file, calculates a 4-character Geohash for every trail (which represents a grid box roughly 24 miles wide), and automatically exports a separate, optimized Parquet file for every grid square.

SQL
INSTALL spatial; LOAD spatial;

COPY (
  SELECT 
    *, 
    -- Create a grouping column using the first 4 characters of the Geohash
    substr(ST_GeoHash(geom), 1, 4) AS region_id 
  FROM st_read('master_state_trails.geojson')
) TO 'terrapulse_regions' (
  FORMAT PARQUET,
  PARTITION_BY (region_id), -- DuckDB will create a separate folder/file for each region
  COMPRESSION 'zstd',
  ROW_GROUP_SIZE 100000
);

Inside your local WebView, you define the startMapDownload function. You will use the modern fetch API and pipe the data directly into an OPFS file handle. This is highly memory-efficient because it streams the data chunk-by-chunk rather than loading a massive 300MB file into the phone's RAM all at once.

JavaScript
async function startMapDownload(url, filename) {
  try {
    // 1. Tell React Native the download started
    window.ReactNativeWebView.postMessage(JSON.stringify({ status: 'started', file: filename }));

    // 2. Open the OPFS directory
    const opfsRoot = await navigator.storage.getDirectory();
    
    // 3. Create the new Parquet file in the sandbox
    const fileHandle = await opfsRoot.getFileHandle(filename, { create: true });
    const writableStream = await fileHandle.createWritable();

    // 4. Fetch the file from your server
    const response = await fetch(url);
    
    if (!response.ok) throw new Error("Network response was not ok");

    // 5. Pipe the network stream directly into the OPFS storage stream
    await response.body.pipeTo(writableStream);

    // 6. Tell React Native it finished successfully
    window.ReactNativeWebView.postMessage(JSON.stringify({ status: 'complete', file: filename }));
    
  } catch (error) {
    window.ReactNativeWebView.postMessage(JSON.stringify({ status: 'error', message: error.message }));
  }
}

Once that pipeTo function finishes, the GeoParquet file is physically sitting on the user's local storage. From there, your DuckDB instance (which is also running in that same WebView) can register the file using db.registerFileHandle() and instantly start querying it for MapLibre, entirely offline.

Here is why explicit downloads are still required for true off-grid use, but obsolete everywhere else:

When Online: The Magic of HTTP Range Requests
If the user is at home, on the highway, or anywhere with a basic cell signal, they do not need to download the map to view it.

Because GeoParquet is a cloud-native format, DuckDB-WASM can read a file that lives on your remote server without downloading the whole file. It uses a web standard called HTTP Range Requests.

When the user pans the map, DuckDB-WASM reads just the metadata headers of your remote file over the network.

It calculates which byte ranges correspond to the trails inside their screen.

It requests only those exact bytes from your server.

The Result: The map streams instantly. They can pan around a 5GB state-wide map file hosted in the cloud, and the app will only use a few megabytes of mobile data.

When Offline: The OPFS Reality
If the user drives out to a desert trail where their phone says "No Service," HTTP Range Requests instantly break down because there is no network to fetch the bytes.

To ensure the map loads in a dead zone, the user must still tap a "Download" button while they are at home or on a strong connection. That download process is what takes the remote GeoParquet file and copies it directly into the phone's local Origin Private File System (OPFS). Once it is physically on the SSD, DuckDB queries the local file instead of the remote server.

The Problem with onX and Gaia (The "Stretch" Effect)
The current scope of legacy apps relies on downloading pre-rendered tile pyramids. Because storing every single zoom level all the way down to the dirt would take hundreds of gigabytes, these companies make a compromise.

They usually cap offline downloads at "Zoom Level 14" or "15".

If you zoom in past that cap to see a tight switchback or read a cluster of topographic contour lines, the app has no new data to show you.

To simulate zooming in, the map engine literally takes the image or vector tile from Zoom Level 15 and stretches it.

The Result: Topographic lines become thick and fuzzy, trail labels become massive and pixelated, and the map becomes almost impossible to read accurately at the micro-level.

The GeoParquet Advantage (Infinite Sharpness)
With the DuckDB-WASM and GeoParquet stack, you are no longer downloading map images or pre-sliced tiles. You are downloading the raw mathematical coordinates of the physical world.

Procedural GPU Rendering: When a user pinches to zoom in on TerraPulse, DuckDB hands the exact mathematical geometry to MapLibre. The device's GPU redraws the lines perfectly from scratch at that exact micro-zoom level.

Zero Pixelation: Because it is rendering math rather than stretching an image, you can zoom in until a trail is 100 feet across on your screen, and the contour lines will remain razor-thin, crisp, and high-definition.
