---
name: Orval header Zod schemas vs Express lowercased headers
description: Why header.safeParse(req.headers) silently fails 100% of the time for OpenAPI header parameters in an Express + Orval + Zod stack.
---

Node/Express always lowercases incoming HTTP header names when populating
`req.headers` (e.g. a client-sent `X-User-Id` arrives as `req.headers["x-user-id"]`).
Orval-generated Zod header schemas, however, keep the exact casing from the
OpenAPI spec's `name:` field (e.g. `zod.object({ "X-User-Id": zod.string() })`).

**Why it matters:** calling `SomeHeaderSchema.safeParse(req.headers)` directly
will fail validation on every single request, since the schema's key never
matches any key actually present on `req.headers`. This fails silently as a
generic 400 rather than an obvious crash, so it's easy to ship and only
discover via live/e2e testing (a plain `tsc` typecheck won't catch it, since
both sides are separately well-typed).

**How to apply:** in any Express route validating an OpenAPI header parameter,
normalize headers into the schema's exact expected shape before parsing —
e.g. `schema.safeParse({ "X-User-Id": req.headers["x-user-id"] })` — rather
than passing `req.headers` straight through. Don't hand-edit the generated
schema (it's regenerated by codegen); fix it at the route handler call site.
